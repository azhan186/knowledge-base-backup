# 知识库规范

> 管理知识库的规则与约定 | 建立于 2026-05-15

---

## 一、目录结构

```
知识库/
├── Raw Sources/              # 原始文件，未经加工
│   └── 上海万豪侯爵酒店资料/  # 酒店 SOP 原始 docx
├── Wiki/                     # 提炼后的知识，可读文档
│   ├── index.md             # 总索引
│   ├── log.md               # 操作日志
│   └── [按分类组织]          # 各主题 Wiki 页面
└── Schema/                  # 结构化数据（待建设）
```

**规则：**
- 原始文件放 Raw Sources，整理后的知识放 Wiki
- 不得将原始 docx / pdf 等直接放入 Wiki 目录
- 每个主题在 Wiki 下有对应的 md 文件作为入口

---

## 二、命名规范

### 文件名
- **原始文件**：保留原始编号 + 原始标题（如 `RM-HSKP-LDY-001制服管理.docx`）
- **Wiki 文件**：使用中文简短标题（如 `制服管理SOP.md`），不得包含 `___` 或数字编号前缀

### Raw Sources 命名
- 原始文件名乱码（传输编码损坏），内容可读的，处理后**必须保留原文件名**（作为元数据），同时生成干净命名的副本
- 禁止在 Raw Sources 目录中保留无法识别的乱码文件名（用干净的复制件替代）

## 三、酒店 SOP 处理流程

1. **接收文件** → 保存到 `Raw Sources/上海万豪侯爵酒店资料/`
2. **读取内容** → 用 python-docx 解析中英文
3. **生成 Wiki** → 转换为 `.md` 文件保存到 `上海万豪侯爵酒店资料/`
4. **记录日志** → 在 `Wiki/log.md` 记一行摘要

### 批量处理（SOP 数量多时）
- 按系列分组：DY（洗衣房）、PA（公共区域）、Floor（楼层）、FC（制服）
- 每组完成后统一通知用户，避免每条都回复 "ok"

## 四、Wiki 页面管理

### 页面状态
- **草稿** `🟡`：刚建立，内容不完整
- **已验证** `✅`：内容经人工核对，无误
- **已过时** `❌`：被新版本替代

### 更新规则
- 每次更新页面，在文件末尾加一行更新日志：
  ```markdown
  ## 更新记录
  - 2026-05-15：xxx
  ```
- 大幅修改时注明更新原因

### 页面清单
- `Wiki页面清单.md` 保持最新，作为所有 Wiki 页面的入口索引
- 新增 Wiki 页面时，同步更新此文件

## 五、沉淀规则

> 什么值得进入知识库？

**应当沉淀：**
- 有知识价值的信息（中英文术语、工作流程、决策逻辑）
- 经过验证的组织结构、岗位信息
- 原创分析成果（如组织架构冲突分析、偏差正常化理论）

**不应沉淀：**
- 纯聊天记录（对话日志交给 memory/）
- 未经核实的推测
- 个人信息（除非用户明确要求）

**沉淀原则：**
> 知识库是给「未来的自己」看的。要经得起一年后回头翻阅。

**主动关联原则：**
收到新知识时，主动思考它属于哪个模块，建议它应与哪些已有页面建立链接。
格式：
```
## 关联页面
- [[相关页面A]]（原因：...）
- [[相关页面B]]（原因：...）
```

## 六、Raw Sources 管理

| 格式 | 是否可直接读取 | 处理方式 |
|------|--------------|----------|
| `.docx`（文字型） | ✅ 可读 | 直接 python-docx 解析 |
| `.doc`（旧格式） | ⚠️  可能损坏 | 尝试转换，失败则跳过 |
| `.pdf`（文字型） | ✅ 可读 | 用 pdf 解析库读取 |
| `.pdf`（扫描图片型） | ❌ 不可读 | 告知用户无法 OCR，跳过 |
| `.xlsx` | ✅ 可读 | 用 openpyxl 解析 |

- 临时文件用 `/tmp/`，不污染工作区
- 发送文件给用户后，保留一份在 workspace 指定目录

## 八、执行原则

> 来自 12 条工程规则的总结，适用于知识库工作

**假设摆台面**
重大假设要明确告知用户，不偷偷假设。不确定就问，不瞎猜。有更简单的方案就提。

**Token 预算**
单任务预算：4,000 tokens。
单会话预算：30,000 tokens。
接近预算时做总结，清理上下文重来，不硬撑。超支要暴露，不掩盖。

**检查点原则**
每完成一个关键步骤：总结做了什么、验证了什么、还剩什么。
状态不清就停下来，不要带着模糊感继续往前走。

**只改该改的**
不顺手 "优化" 旁边的内容。没坏就不重构。遵循现有文件的风格和结构。

**暴露问题，不掩盖**
如果跳过了某条记录、某个文件、某个边界情况，明确说出来。
做不到就说做不到，不要假装一切都好。

**以终为始**
先定义 "做成了" 是什么样子，迭代直到验证通过。不是告诉 AI 走哪几步，而是告诉它终点长什么样。

## 九、Wiki 链接规则（核心目的）

> **Wiki 的价值在于链接，不在于堆文件。**

**目的：**
各类知识文件之间建立联系，通过链接在不同文件之间跳转，形成知识网络，以便在回答问题时能快速调用相关联的上下文。

**具体做法：**

1. **每篇 Wiki 页面必须包含相关链接**
   - 在页面末尾或相关术语处，链接到其他相关的 Wiki 页面
   - 链接格式：`[[相关页面名]]`（飞书文档格式）或 Markdown 链接 `[页面名](相关页面.md)`

2. **链接粒度**
   - **术语链接**：首次出现相关概念时加链接（如「洗衣房SOP → 制服管理SOP」）
   - **底部关联区**：每页末尾加「关联页面」小节，列出3-5个最相关的 Wiki 页面
   - **双向链接**：如果页面A链接到页面B，页面B也应链接回页面A（尽量保持对称）

3. **不得孤立存放文件**
   - 新建的 Wiki 页面必须链接到至少一个已有页面
   - 已有页面被大量引用时，检查是否需要补充反向链接

4. **链接质量优先于数量**
   - 相关性低的链接不如不链
   - 优先链接高频被查询的关联内容

**关联页面的判断标准：**
- 同一个项目内的不同文档 → 链接
- 同一领域的不同主题 → 链接
- 因果关系或上下级关系（如 SOP → 组织架构）→ 必须链接
- 互相印证或对比的内容 → 链接

## 十、Wiki 文件命名规范（重要）

> **知识库文件名为中文，破折号统一。** 本节为 2026-06-11 重大改动后的新约定。

### sources/ 文献摘要命名

**唯一标准格式**：`source-YYYY-MM-DD-中文名.md`

- 统一使用破折号 `-`（不是下划线）
- 统一以 `source-` 为前缀
- 日期前缀 `YYYY-MM-DD`
- 中文部分用破折号或空格连接，不加下划线

**示例：**
- ✅ `source-2026-05-27-中国能否复制华为式跨越.md`
- ✅ `source-2026-06-10-RAID入门篇.md`
- ❌ `2026-06-10_token-deep.md`（纯英文）
- ❌ `source-2026-06-05_黑天鹅事件.md`（下划线连接）
- ❌ `source-2026-06-05-2026-06-05.md`（重复日期）

### 必要时保留英文术语

如果主题英文术语是行业标准/唯一识别，在中文后保留：
- `source-2026-06-08-NASA_MAVEN任务终结.md`（NASA 为领域术语）
- `source-2026-06-10-RAID入门篇.md`（RAID 为行业缩写）

如果同主题有多个变体，用副标题区分：
- `source-2026-06-05-股票资深篇_分析框架.md`
- `source-2026-06-05-股票资深篇_预期差定价.md`

### Raw Sources 命名

原始素材中文名即可，不需 `source-` 前缀：
- `Raw Sources/AI与科技/2026-05-27_中国能否复制华为式跨越.md`

### 概念/实体卡命名

- 直接中文名，不加日期、不加前缀
- `wiki/concepts/光刻机.md`
- `wiki/entities/中芯国际.md`

## 十一、重复内容处理流程（重要）

> **wiki/sources/ 出现同名主题的多个版本 = 重复，必须处理。** 本节为 2026-06-11 处理 6/5 系列重复后的总结。

### 重复判断原则

两个或多个文件满足以下任一条件即为重复：

1. **同名同主题**：文件名仅分隔符不同（`_` vs `-`），且主题一致
2. **重复引用**：多个文件被同一组上游文件引用，引用形式几乎相同
3. **内容高重叠**：标题相同、章节结构相同，只是细节略有差异

### 处理流程

1. **识别**：扫描 `wiki/sources/` 下所有文件，按主题分组
2. **判断保留哪个**：
   - 内容全面性（覆盖更多主题维度）
   - 更新时间（最新优先）
   - 上游引用计数（被引用多者保留）
3. **保留一个，其余迁移至 trash**：
   ```bash
   mkdir -p _trash/<date>_<原因>/
   git mv <重复文件> _trash/<date>_<原因>/
   ```
4. **更新上游引用**：
   - 所有 `[[旧名]]` 改为 `[[新名]]`
   - 检查并更新 `topics-index.md` 中重复条目
5. **验证**：`validate-snapshot.sh` 检查产物完整性

### 不动的情况下

- 内容完全不同主题（即使文件名相似）→ **保留两个**、加副标题区分
- 主卡/概念卡引用但文件不存在 → **记录缺口**，后续可新建

### trash 保留策略

`wiki/sources/_trash/` 下保留删除文件的本地副本 30 天，确认无问题后可手动清理。**不推荐直接 `rm -rf`**，因为 git 历史可能遗漏某些改动。

## 十二、链接解析与别名约定

> **Obsidian [[双向链接]] 是知识库索引的核心机制。** 本节明确链接写法约定，避免脚本检测不到的"孤儿链接"。

### 三种正确写法

```markdown
1. 裸文件名：[[光刻机]]
2. 别名形式：[[源文件名|起个能看懂的别名]]  ← 推荐
3. 带锚点：[[光刻机#EUV部分]]
```

### 推荐使用别名

为防止 Obsidian 全局搜索歧义，同一主题多个版本时优先使用别名：

```markdown
[[楼层SOP|万豪侯爵楼层SOP体系]]
[[万豪侯爵酒店客房部组织架构|万豪侯爵客房部组织架构]]
```

### 不允许的写法

- 别名符号 `|` 前后有空格：`[[光刻机 | EUV]]` → Obsidian 不识别
- 日期中间用空格：`[[2026-06-10 RAID入门篇]]` → 空格破坏了文件名匹配
- 别名取代源名：`[[中文描述]]` 而无对应文件 → 无法解析

### 验证脚本

```bash
bash scripts/validate-snapshot.sh  # 检查 snapshot 一致性 + 产物完整性
```

### 已知缺口（截至 2026-06-11）

约 400+ 个 [[]] 链接指向尚不存在的主卡/概念卡（太阳、RAID6、机械硬盘、处置效应、信息熵、引力波等）。这些是**知识内容缺失**，不是链接错误。后续可逐步新建概念卡消除。

## 十三、三层架构（来自 Schema 版）

> 知识库的三层职责分工

### 第一层：Raw Sources（原始资料）
- **定位**：只属于用户的所有原始资料
- **权限**：**只读**，不可修改（保持事实基准）
- **内容**：用户提供的一切原始文件、聊天记录、文档等
- **存储位置**：`图书馆/Raw Sources/`、`新闻/Raw Sources/`

### 第二层：Wiki（知识库）
- **定位**：由我负责维护的知识库
- **职责**：
  - 从 Raw Sources 中读取、摘抄
  - 交叉对比验证信息
  - 整理、归纳、记录
  - 保持信息的准确性和时效性
- **存储位置**：`wiki/`（统一顶层，详见第一节）

### 第三层：Schema（规则文件）
- **定位**：配置文档，用户和我共同维护
- **内容**：Wiki 的结构规范、命名约定、工作流程、更新机制
- **存储位置**：`wiki/syntheses/知识库规范.md`（**统一权威**，原 Schema 版已废弃）

3. **备份/迁移单扫**：单一扫描路径
4. **索引统一**：一个 wiki/index.md 服务全库

**保留 AGENTS.md 的 Raw Sources 分类规则**（AGENTS.md 和 P2-a 一致）：
- `图书馆/Raw Sources`：知识性文章、AI教程、科学原理、理论分析
- `新闻/Raw Sources`：时事新闻、财经新闻、国际新闻
