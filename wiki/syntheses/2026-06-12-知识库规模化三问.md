## Q1: 性能随节点增长的演进

### 📊 现状分析

| 节点规模 | SVG 渲染 | 力模拟 | 交互响应 | 内存 |
|---------|---------|--------|---------|------|
| 750（当前） | 60 fps | 1-2s 收敛 | 流畅 | 30MB |
| 2,000 | 45-55 fps | 3-5s 收敛 | 偶发卡顿 | 80MB |
| 5,000 | 20-30 fps | 10s+ 收敛 | 明显卡顿 | 200MB |
| 20,000 | 5-10 fps | 30s+ 收敛 | 不可用 | 800MB |
| 100,000 | <5 fps | 永远不收敛 | 崩溃 | 4GB+ |

**关键瓶颈**：
1. **DOM 元素数**：每个 SVG text/circle 都是 DOM 节点，> 3000 时浏览器显著减速
2. **力模拟**：O(N²) 计算每帧位置
3. **重绘成本**：paint-order stroke + halo + 1682 lines = 多次重绘
4. **pointer events**：751 hover listener 触发频繁

### 🛠 三阶段演进方案

#### Phase 1（节点 0-5,000）— "Canvas + Web Worker + 智能 culling"

**目标**：把 SVG 替换为 Canvas 2D + Web Worker 跑力模拟

**技术栈**：
- **Canvas 2D** 替代 SVG（性能 +5-10x）
- **d3-quadtree** 做视口裁剪（只渲染可见节点）
- **Web Worker** 跑力模拟（不阻塞 UI）
- **OffscreenCanvas**（如果浏览器支持）
- **d3-force** 异步驱动

**改动范围**：
- 替换 index.html 中 SVG → Canvas 渲染层
- 节点/边数据通过 postMessage 在 worker 传递
- 标签仍然 SVG（标签数量本身经过 LOD 控制，性能 OK）

**工作量**：3-5 天
**预期收益**：5,000 节点 60fps

#### Phase 2（节点 5,000-30,000）— "WebGL + 预计算聚类"

**目标**：WebGL 渲染 + 预计算多级聚类

**技术栈**：
- **PixiJS** 或 **regl** 做 WebGL 渲染
- 4-5 级预计算聚类（zoom 0.1, 0.3, 0.5, 1.0, 2.0）
- 平滑插值切换层级
- 标签仍用 SVG/HTML（可单独 LOD）

**预计算聚类**：
- 离线脚本定期生成 `clusters-lod.json`
- 每级包含：{cellId, nodeCount, dominantLabel, importance, position}
- 浏览器只渲染当前 zoom 对应层级

**改动范围**：
- 增加 `wiki-graph/build-clusters.py`（离线）
- 替换 Canvas → WebGL 渲染器
- 加 zoom level 切换逻辑

**工作量**：1-2 周
**预期收益**：30,000 节点 60fps

#### Phase 3（节点 30,000+）— "Tile-based + 服务端索引"

**目标**：视口分块加载 + 服务端搜索

**技术栈**：
- 类似地图的 tile 系统
- 后端 API 索引（SQLite FTS5）
- 客户端只持有当前视野的节点
- 搜索/详情走 API

**改动范围**：
- 改造为 Node.js/Python 后端
- 数据库存储
- 客户端薄壳

**工作量**：1 个月
**预期收益**：可扩展到 100 万节点

### 🎯 我的建议

**现在到 Phase 1**（优先）：
- 节点到 2,000 时启动
- 投入小、收益大
- 不改变用户交互模式

**Phase 2 等节点过 5,000 再启动**
**Phase 3 等知识库真正成为「主力工具」再考虑**（需要服务端 + 多人协作）

## Q2: 词条分级 + 缩放聚类同步

### 🎯 当前问题

当前聚类**只看空间位置**：
- `gridSize = 100 / scale`
- 每个网格内的节点合并
- 球上只写**数量**（如 "12"）

这导致：
- 不知道球里是**什么**
- 重要节点**也会被合并**（不尊重重要性）
- 微不足道的孤立节点在 0.85-1.0 zoom 还**显示着**

### 📐 词条分级方案

#### 4 个维度 + 公式

```
importance = 0.40 * normalize(backlinks)
           + 0.25 * isMainCard
           + 0.20 * typeWeight
           + 0.10 * recencyBoost
           + 0.05 * masterRating
```

| 维度 | 说明 | 取值 |
|------|------|------|
| **backlinks** | 多少节点引用我 | `log(1 + backlinks) / log(1 + maxBacklinks)` |
| **isMainCard** | 主人是否标记为主卡 | 1.0 / 0.0 |
| **typeWeight** | 节点类型权重 | concept=1.0, synthesis=0.8, entity=0.5, source=0.3 |
| **recencyBoost** | 最近更新加成 | `max(0, 1 - daysSinceUpdate/365)` |
| **masterRating** | 主人手动评分（未来） | 0-1 |

**数据来源**：
- backlinks：已有（graph.json 边的反向统计）
- isMainCard：从 frontmatter 的 `main: true` 字段读
- typeWeight：从 path 推断（已有）
- recencyBoost：需要 frontmatter 加 `last_updated` 字段
- masterRating：未来主人可加 `priority: 1-5`

#### 输出范围

`importance ∈ [0, 1]`

- 0.0-0.2：低（孤立来源、临时笔记）
- 0.2-0.5：中（一般实体、一般来源）
- 0.5-0.8：高（重要概念、综合笔记）
- 0.8-1.0：核心（主卡、高连接度概念）

### 🧮 智能聚类算法

#### 替换当前的"网格硬聚类"

```python
def cluster_at_zoom(nodes, viewport, scale):
    grid_size = 100 / scale  # 屏幕 100px 单元
    
    for each cell in grid:
        cell_nodes = [n for n in nodes if in_cell(n, cell)]
        if not cell_nodes:
            continue
        
        # 1. 挑出"必须单独显示"的节点
        individual_threshold = zoom_to_threshold(scale)
        # scale=2.0 → 0.0 (全显)
        # scale=1.0 → 0.7
        # scale=0.6 → 0.85
        # scale=0.4 → 0.95
        must_show = [n for n in cell_nodes if n.importance > individual_threshold]
        cluster_me = [n for n in cell_nodes if n.importance <= individual_threshold]
        
        # 2. 渲染 must_show 为单点
        for n in must_show:
            render_node(n)
        
        # 3. cluster_me 合并为一个球
        if cluster_me:
            render_cluster(
                position=avg(cluster_me),
                count=len(cluster_me),
                total_importance=sum(n.importance for n in cluster_me),
                top_label=argmax(cluster_me, key=importance).displayName,
                color=dominant_color(cluster_me)
            )
```

#### 球的信息升级

**当前**：`<big circle> 12`（只显示数量）

**新设计**：`<big circle> RFID +11`
- 显示**最高重要度节点的显示名**（truncate 8 字符）
- 后面加 `+N` 表示还有 N 个
- 球大小 = `8 + √total_importance × 10`（用重要性，不是数量）
- 颜色 = 最高重要度节点的颜色

**hover tooltip**：
```
RFID 射频识别（+ 11 节点）
  - RFID 不可能三角 (0.92)
  - 反向散射调制 (0.71)
  - 柔性 RFID 洗涤标签 (0.68)
  - ...还有 8 个
```

### 📊 缩放阈值表

| zoom | 必显阈值 | 显示数量级 | 体感 |
|------|---------|-----------|------|
| 2.0+ | 0.0 | 全部单点 + 全标签 | 细节查看 |
| 1.0-2.0 | 0.6 | 重要单点 + 聚类 | 浏览 |
| 0.6-1.0 | 0.8 | 极重要单点 + 聚类 | 概览 |
| 0.4-0.6 | 0.95 | 几乎全聚类 | 鸟瞰 |
| <0.4 | 1.0 | 仅聚类 | 概览图 |

### 🎯 实施步骤

1. **数据层**：graph 生成时计算 importance（precompute）
2. **前端层**：更新 `getColor` / `updateClusters` 使用 importance
3. **UI 层**：球的 label 升级为 `displayName +N`
4. **展示层**：legend 加"按重要性浏览"模式

**工作量**：3-5 天
**预计收益**：聚类从"模糊数量"升级为"语义浏览"

## Q3: 高效编辑 / 校对 / 扩展工作流

### 🎯 当前痛点

1. **写**：在 VSCode 写 .md → 保存 → 跑 `mark-compiled.sh` → 浏览器强刷
2. **查**：搜索节点 → 找到 → 侧栏看内容预览 → 复制路径 → VSCode 打开
3. **校**：无机制（没有"过期检查"、"断链检查"、"未引用检测"）
4. **扩**：写新内容时不知道**该引用谁**、**谁该引用我**
5. **改**：双向链接改名字要**手动找出所有引用者**同步改

### 🛠 三个层次方案

#### Layer 1（2 周）— "低摩擦工作流"

**目标**：减少"写完忘编译"、"编译完忘强刷"这类基础摩擦

**新增 CLI 工具**（在 `scripts/`）：

```bash
# 创建新词条（自动 stub、frontmatter、加入待编译队列）
wiki-new "AI 幻觉专家篇" --type=source --tags=AI,深度学习

# 检查双向链接
wiki-check-links          # 找出所有 [[xxx]] 失效
wiki-check-stale          # 6 个月没更新的页面
wiki-check-orphans        # 0 反向链接的孤立页面
wiki-check-missing-meta   # 缺 frontmatter 的页面

# 自动链接建议（基于 tag 相似度）
wiki-suggest "AI幻觉"     # 输出"建议链接到：深度学习、大语言模型、概率论"

# 编译 + 通知
wiki-compile --watch      # 文件改动自动编译
```

**Live Reload**：
- 加 `wiki-graph/auto-reload.html` 或在 index.html 里加 `EventSource`
- graph.json 改动时自动重载（无需强刷）
- 改文件 → 1 秒内图谱更新

**工作量**：1 周
**预期收益**：写 → 看到结果的延迟从 30s 降到 1s

#### Layer 2（1 个月）— "Web 版编辑器"

**目标**：在浏览器里直接编辑，零切换

**功能**：
- **Monaco Editor**（VSCode 同款内核）嵌入浏览器
- **Markdown 实时预览**（CodeMirror 也行）
- **`[[双向链接]]` 自动补全**（输入 `[[` 弹出候选）
- **Frontmatter 表单化**（不用手写 YAML）
- **保存按钮** → 调 API 写文件（或下载 .md 让你手动 commit）

**架构**：
- 轻量后端：`/home/azhan186/open claw/wiki-graph/api.py`（FastAPI）
  - `POST /api/file` — 写文件
  - `GET /api/file?path=...` — 读文件
  - `GET /api/suggest-links` — 推荐链接
- 鉴权：主人 LAN 内访问，可加 Cloudflare Access 公开时启用
- 编辑器：单 HTML 文件 + Monaco CDN

**工作量**：2 周
**预期收益**：完全在浏览器内完成所有操作

#### Layer 3（3 个月）— "知识库即平台"

**目标**：可扩展、可协作、可版本化

**架构升级**：

```
┌─────────────────────────┐
│   浏览器（Web Editor）    │
└───────────┬─────────────┘
            │ REST/GraphQL
┌───────────▼─────────────┐
│   API 层（FastAPI）       │
├─────────────────────────┤
│   核心引擎                │
│  ├ 链接解析              │
│  ├ 冲突检测              │
│  ├ 重要性评分            │
│  ├ 聚类预计算            │
│  └ 全文搜索 (FTS5)       │
├─────────────────────────┤
│   存储                   │
│  ├ Markdown 文件 (git)   │
│  ├ SQLite 索引 (FTS5)    │
│  └ 嵌入向量 (可选)       │
└─────────────────────────┘
```

**插件系统**：
- 核心只提供 hooks（`onSave`, `onCompile`, `onSearch`）
- 新功能 = 新插件
- 例：日历视图插件 = `onSearch('date:2026-06-12')`
- 例：闪卡插件 = `onSearch('type:flashcard')`

**AI 辅助**：
- 写新主题时，AI 草拟 + 主人编辑
- 自动找出可链接的邻居
- 自动检查信息矛盾
- 自动提取关键事实到主卡

**Git 集成**：
- 每次保存自动 commit
- Diff 视图
- 历史回滚
- 多人协作（未来）

**工作量**：3 个月
**预期收益**：知识库成为"会成长的工具"而非"死的文件"
