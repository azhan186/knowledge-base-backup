# 词元（Token）深入篇

## 一个 token 是如何诞生的：BPE 算法拆解

模型怎么"知道"要把 "unbelievable" 切成 "un" "believe" "able"？这不是靠人工规则，而是靠自下而上的统计算法——最主流的就是字节对编码（BPE）。

### 虚构极小例子模拟核心步骤

假设语料库只有三句话："low"、"lower"、"widest"。

**第一步：初始化**
把每个字符都当成一个独立的 token，包括空格和末尾符号（用 _ 表示词尾）。初始词表就是所有单个字符：l, o, w, e, r, i, d, s, t, _。

**第二步：反复合并**
不停地扫描整个语料，找出出现频率最高的相邻 token 对，把它们合并成一个新的 token 加入词表。

- 第一轮：统计所有相邻对，"l"和"o"出现2次（来自 low, lower），假设频率最高的是 "l"+"o" → "lo"。合并后词表多了 "lo"。
- 第二轮："lo"+"w" 又成了最频繁对，合并出 "low"。
- 持续进行，直到达到预设词表大小（如50,000个token）。

最终，词表里会同时存在 low、er、est、wid 等片段。遇到新单词 "lowest"，即使从未整体出现过，分词器会切成 low + est + _。尽管模型没见过整个词，但见过零件，就能大概推测其含义。

**这种方法的神奇之处**：能自动发现词根、前缀、后缀，对任何语言都管用，完全不用懂语言学。

【可靠度评价】BPE 算法的基本流程是成熟且公开的共识知识，例子中虚构了语料和数字，但算法逻辑是准确复现的，可靠度很高。

## 深入理解词元化的影响：牵一发而动全身

### 权衡一：序列长度与专注范围

模型的"注意力"一次能处理的 token 数量是固定的，叫上下文窗口。

- 词级分词：遇到生词就会宕机
- 子词级分词：中文"国际化"可能被切成"国""际""化"三个token，比词级多用2个位置。整篇文章下来 token 总数可能膨胀 30%-50%，有效上下文窗口被压缩。本来能一次看完十章小说，现在只能看六章。

### 权衡二：计算开销与模型容量

假设输入有 N 个 token，Transformer 架构的自注意力计算开销大致与 N 的平方成正比（计算量 ≈ N × N 的某个倍数）。token 数量增加不仅仅是被"拉长"，运算量会平方级增长。这也是为什么不能无限制地把字拆小——字符级分词的 N 大得离谱，训练和推理成本都会飙升。子词级分词成为无奈且最优的折中。

### 权衡三：语义密度的巨大差异

不同语言的"信息密度"不同：
- 一个中文成语四个字可能代表一个复杂典故，却被切成4个token
- 英文可能要用一整句话表达，被切成10个token

模型内部却可能只对英文那几个 token 的算术组合更"熟悉"（因为训练数据里英文占比高）。这就导致用中文交流时，可能被迫用更多字去解释同一个概念，白白消耗 token 配额。

### 权衡四：分词器和模型是"连体婴"

分词器一旦训练好，词表（如32,000个id）就和模型的第一层权重死死绑定。如果觉得某个术语没切好（如 "ChatGPT" 被切成 "Chat" "G" "PT"），想调整，对不起，整个模型得重新预训练，成本无法接受。分词器的设计缺陷会伴随模型整个生命周期。

【可靠度评价】平方复杂度、序列长度与上下文的权衡、模型与分词器绑定的现状，均是当前 Transformer 架构下的公认事实和常见折衷，可靠度很高。语义密度带来的公平性争议也属于普遍认知。

## 前沿：更深细节与可能的未来

### 1. 数字与推理的"魔咒"

子词分词对数字的伤害远比想象的深：
- 同一个数值 "42" 可能被切成 "4" 和 "2"，模型看到的不是数字，而是两个毫无数值关联的符号
- "421" 可能被切成 "42" 和 "1"，模型完全没有学到 "42" 和 "421" 之间的十进制进位关系

有人尝试把数字拆成每个数位（如 "421" 变成 "4""100""2""10""1"），让模型看到位值。

【不确定性】这些方法仍处在实验阶段，没有哪一种被广泛证明能彻底解决大模型的算术难题。目前一些模型能通过代码解释器绕开 token 来算数，但这属于外挂，并非模型自身的理解。

### 2. 多模态"统一词元"的尝试

现在流行的多模态大模型，常把图像切成很多个固定大小的小方格（patch），每个小方格被一个视觉编码器变成一个向量——这个向量就是一个视觉词元。

**新问题**：
- 语言 token 是离散的（如 id 378），而视觉 token 往往是连续的向量
- 为了把它们放进同一个模型：要么把视觉 token 离散化（又损失信息），要么给模型设计特殊的结构来混合处理
- 一个 224×224 的图片切出来可能有几百个视觉 token，与文字 token 的计算资源分配极不均衡

### 3. 彻底抛弃 token 化？字节级模型之路

一个激进的前沿方向：能不能不切分任何东西，直接让模型一个字节一个字节地读取原始 UTF-8 编码？

- 文本 "cat" 变成 63 61 74 三个字节的序列
- **巨大诱惑**：天然跨语言、零集外词、能完美拼写、不受拼写错误干扰
- **代价**：序列会变得极长（是子词 token 的 4 倍以上），计算量按平方增长，训练极其困难

【不确定性】目前已有一些阶段性成果（如某些字节级模型架构），成功降低了长序列计算的复杂度，使得字节级建模变得可行。但它们在同等计算量下的性能，相比顶尖子词级模型仍有差距。这个方向可能是未来的终极形态，但现在还在路上。

【可靠度评价】数字切分的危害和字节级模型的基本动机是共识。但关于"如何让模型真正理解数字"以及"字节级模型能否替代子词模型"的问题，目前没有公认的最佳答案，属于活跃的前沿争议。
